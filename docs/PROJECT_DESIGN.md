# Kwiz — Project Design Document

> **TUI Retro · AI-Powered · Quiz Maker**  
> *Exam sheets, but make them terminal.*

---

## Vision

Kwiz is a **local-first** quiz maker. Quizzes are generated by your PC (via `llama.cpp`), stored on-device in AsyncStorage, and work fully offline. Google Drive acts as a **sync courier** — not a database — ferrying quiz JSON between your devices. `ntfy.sh` WebSockets signal the phone to pull when new content arrives. No cloud AI costs. No mandatory sign-in. Pure retro TUI vibes.

> [!IMPORTANT]
> **Local-first rule**: Every quiz is always readable from local storage first. Google Drive is additive — it provides sync and backup. The app must function fully without a Drive connection.

---

## Aesthetic Direction

The UI follows the **TUI Retro** visual language from `Mobile/src/components/` — meaning the *look* is inspired by terminal UIs (hard borders, monospace, no softness), but the *content* is a normal clean app. No terminal jargon, no hacker copy, no `>_` prompts.

| Token | Value |
|---|---|
| Font | `JetBrains Mono` (400 + 700) |
| Borders | Hard pixel borders, no border radius |
| Shadows | `4px 4px 0px` hard drop shadows |
| Palette | High-contrast dark/light modes |
| Animations | Subtle transitions, skeleton pulse |
| Labels | Title case, bracket-wrapped e.g. `[ Label ]` |
| Badges | Solid-fill accent chips |

**What this is NOT:** No scanlines, no blinking cursors, no "SYSTEM READY" copy, no terminal prompts, no fake loading sequences. Just a clean, structured, retro-styled UI.

Existing components already available:
- `TuiButton`, `TuiContainer`, `TuiDrawer`, `TuiHeader`
- `TuiInput`, `TuiText`, `TuiCheckbox`, `TuiSwitch`
- `TuiCalendar`, `TuiSkeletonLoader`, `TuiTabBar`
- `SplashIcon`

## Quiz Data Schema

```ts
// A single question in a quiz
type QuizQuestion =
  | {
      type: 'multiple_choice';
      question: string;
      choices: string[];          // e.g. ["A. ...", "B. ...", "C. ...", "D. ..."]
      answer: string;             // The correct choice letter/text
    }
  | {
      type: 'identification';
      question: string;
      answer: string;             // Exact answer string
      charCount: number;          // len(answer) — drives the letter-box UI
    };

// A complete quiz set
type QuizSet = {
  id: string;                     // uuid
  title: string;                  // LLM-generated or user-defined
  subject?: string;
  createdAt: string;              // ISO timestamp
  questionType: 'multiple_choice' | 'identification' | 'hybrid';
  questions: QuizQuestion[];
  source?: string;                // 'attachment' | 'prompt'
  status: 'generating' | 'ready' | 'error';
};
```

### LLM Output Format (compact — "maytoon")
To keep token usage minimal, the LLM outputs a compact JSON shorthand:

```json
{
  "t": "Photosynthesis Quiz",
  "qs": [
    { "k": "mc", "q": "What gas do plants absorb?", "c": ["O2","CO2","N2","H2"], "a": "CO2" },
    { "k": "id", "q": "The pigment that captures light is called ___.", "a": "chlorophyll", "n": 11 }
  ]
}
```
The mobile app parses `k: "mc" | "id"` and expands into the full `QuizQuestion` shape. `n` is the character count for identification letter-boxes.

> **Goal**: The Tauri desktop app accepts a file/attachment or text prompt, calls the local `llama.cpp` server, and outputs a quiz JSON.

- **AI Backend**: `llama.cpp` running locally, exposed as an OpenAI-compatible HTTP API (`http://localhost:8080/v1/chat/completions`)

### Phase 3 — Integration (Google Drive + ntfy.sh)

> **Goal**: Desktop generates a quiz → uploads to Google Drive → ntfy.sh pings the phone → phone pulls and displays it locally. Both devices remain fully functional without Drive.

#### Architecture: Local-First + Drive as Courier

```
┌─────────────────────────────────────────────────────────┐
│                  SOURCE OF TRUTH                        │
│           AsyncStorage  (on each device)                │
└────────────────────────┬────────────────────────────────┘
                         │
          Google Drive (sync courier only)
                         │
          ┌──────────────┴─────────────┐
          │                            │
    PC writes quiz               Phone reads quiz
    to Drive                    from Drive → saves
          │                     to AsyncStorage
          ▼
   ntfy.sh POST "sync"
          │
          ▼
   Phone WS receives → pulls from Drive
```

**Key rules:**
- Quizzes are always saved to AsyncStorage **first** before any Drive upload
- App works fully offline — Drive upload/pull is best-effort
- Drive is never queried on quiz playback — always reads local
- If Drive is unavailable, quizzes stay local and sync later

#### Sync Flow (borrowed directly from BootHub)

```
PC generates quiz → saved locally
       │
       ▼
Upload .kwiz.json to Google Drive  (Kwiz/Quizzes/)
       │
       ▼
POST https://ntfy.sh/kwiz-sync-{hexEmail}  body: "sync"
       │
       ▼  (WebSocket already open on phone)
Phone WS onmessage → pullNewQuizzesFromDrive()
       │
       ▼
New .kwiz.json files fetched → merged into AsyncStorage
       │
       ▼
Skeleton card resolves → quiz appears ✓
```

#### Sync Engine (`Mobile/src/utils/sync-engine.ts`)

Lifted directly from `BootHub/src/utils/sync-engine.ts` and `google-drive.ts`, adapted for quiz data:

```ts
// Topic derivation (same as BootHub)
const hexEmail = email.split('').map(c => c.charCodeAt(0).toString(16)).join('');
const topic = `kwiz-sync-${hexEmail}`;
const wsUrl = `wss://ntfy.sh/${topic}/ws`;

// Phone listens:
ws.onmessage = (event) => {
  const data = JSON.parse(event.data);
  if (data.event === 'message' && data.message === 'sync') {
    pullNewQuizzesFromDrive();
  }
};

// PC publishes after upload:
await axios.post(`https://ntfy.sh/${topic}`, 'sync', {
  headers: { 'Content-Type': 'text/plain' },
});
```

## Open Questions

> [!IMPORTANT]
> **Attachment handling on mobile**: Should the phone *also* be able to send attachments (camera photos of notes) to the desktop for LLM processing? Or is the desktop the only generation endpoint?

> [!IMPORTANT]
> **Offline-first quiz taking**: Should quizzes be fully playable offline once synced? (Assuming yes — just confirming.)

> [!NOTE]
> **Maytoon shorthand**: The compact JSON format (`k`, `q`, `c`, `a`, `n`, `t`, `qs`) is internally called "maytoon" per your brief. Should this be documented as a named format or just kept as an implementation detail?

> [!NOTE]
> **Quiz editing**: Phase 1 scope — should users be able to edit/rename a quiz after generation, or is that a later phase?

> [!NOTE]
> **Multiple quiz sessions**: Should the app track per-quiz scores/history across multiple attempts, or just show the last result?
